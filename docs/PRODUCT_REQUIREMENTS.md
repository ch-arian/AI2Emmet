# Product Requirements Document (PRD)
## Product: emmet-ui
## Version: 1.0 (Derived from current codebase)
## Date: March 25, 2026

---

## 1) Executive Summary

`emmet-ui` is a React library that converts compact Emmet abbreviations (often generated by an LLM) into secure, interactive React UI trees. The core product value is to enable fast AI-driven UI generation with significantly lower token usage than JSON-based UI schemas, while preserving safety (XSS sanitization), extensibility (component registry), and interaction patterns (data-action callbacks).

In practical terms, this product allows a host application to:
- Ask an LLM for an Emmet string,
- Render that response as real UI,
- React to user interactions through an action callback,
- Loop back to the LLM with action context to generate the next UI state.

---

## 2) Problem Statement

### 2.1 User Problem
Teams building LLM-powered experiences need a dynamic UI contract that is:
- More token-efficient than verbose JSON UI schemas,
- Easier to constrain than free-form HTML,
- Safer than directly rendering arbitrary model output,
- Integratable with React component systems.

### 2.2 Product Problem
Without a structured rendering pipeline, teams risk:
- Unsafe content injection (XSS),
- Inconsistent UI behavior,
- Difficult debugging and maintenance,
- Poor interoperability with existing React component libraries.

---

## 3) Product Goals and Non-Goals

### 3.1 Goals
1. Convert Emmet input into React output through a predictable pipeline.
2. Enforce strong HTML sanitization before any hydration step.
3. Support custom component substitution via a registry.
4. Support action handling via declarative `data-action` attributes.
5. Provide both headless (hook) and batteries-included (component) integration APIs.
6. Ship TypeScript-friendly APIs and reusable building blocks.

### 3.2 Non-Goals
1. Not a full design system.
2. Not a visual drag-and-drop builder.
3. Not a server-side policy engine for prompt safety.
4. Not a comprehensive HTML renderer (explicit allowlist intentionally narrows scope).

---

## 4) Target Users and Personas

### 4.1 Primary Persona: AI App Engineer
- Builds React apps with LLM workflows.
- Needs deterministic rendering from model outputs.
- Wants easy action feedback loop (`onAction`) to call model again.

### 4.2 Secondary Persona: Front-End Platform Engineer
- Owns component consistency and safety.
- Needs centralized registry control for which tags are allowed/mapped.
- Needs predictable extension points and testability.

### 4.3 Tertiary Persona: Product Prototyper
- Wants quick iteration with minimal schema overhead.
- Needs usable default components (button/card/text/container) out-of-box.

---

## 5) Product Scope

### 5.1 In Scope
- Emmet parsing to HTML.
- HTML sanitization with explicit allowlist configuration.
- HTML-to-React hydration.
- Component replacement by tag via registry.
- Action delegation (`data-action`, optional JSON `data-payload`).
- Error and loading states for top-level renderer.

### 5.2 Out of Scope
- Rich form orchestration (validation engines, form state managers).
- Accessibility auditing beyond what React/native semantics provide.
- Animation orchestration.
- i18n and localization frameworks.

## 6) Functional Requirements

### FR-1: Emmet Expansion
**Requirement:** System MUST accept a non-empty Emmet abbreviation string and expand it to HTML.

**Behavioral details:**
- Empty or whitespace-only input is rejected with an error.
- Expansion failures return descriptive errors.
- Output is compacted (formatting whitespace removed).

**Acceptance criteria:**
- Given valid Emmet input, output HTML string is returned.
- Given empty input, an exception indicates invalid empty abbreviation.

---

### FR-2: HTML Sanitization (Security-Critical)
**Requirement:** System MUST sanitize all expanded HTML before hydration.

**Behavioral details:**
- Use explicit tag allowlist.
- Use explicit attribute allowlist.
- Block generic data attribute pass-through except specific allowlisted keys.
- Preserve safe text content when unsafe tags are removed.

**Acceptance criteria:**
- Script/event-injection vectors are removed.
- `data-action` and `data-payload` survive sanitization.
- Unsupported tags/attrs do not survive.

---

### FR-3: Hydration to React
**Requirement:** System MUST convert sanitized HTML into React elements.

**Behavioral details:**
- Hydration supports parser options with replace callbacks.
- Non-element nodes pass through.
- Output type may be element, array, or text node depending on input.

**Acceptance criteria:**
- Given safe HTML, React-renderable output is produced without manual parsing by consumer.

---

### FR-4: Component Registry Mapping
**Requirement:** System MUST support mapping tag names to React components.

**Behavioral details:**
- Registry supports `register`, `get`, `has`, and immutable `getAll` copy retrieval.
- Default registry includes common tags/components.
- Replacement callback recursively transforms child nodes.

**Acceptance criteria:**
- Registered custom tags render mapped components.
- Unmapped non-HTML tags render fallback `UnknownTag` component.

---

### FR-5: Action System
**Requirement:** System MUST support declarative click actions via `data-action` and optional `data-payload`.

**Behavioral details:**
- Single delegated click handler on root container.
- Traverses upward through nested actionable ancestors.
- Payload attempts JSON parse; invalid JSON returns an Error object (non-throwing).

**Acceptance criteria:**
- Clicking actionable element calls consumer callback with action + payload + event.
- Nested actionable structures trigger in bubbling walk order.

---

### FR-6: Public Rendering APIs
**Requirement:** Product MUST expose:
1. `pipeline()` for low-level composition,
2. `useEmmetRenderer()` for headless stateful hook usage,
3. `EmmetRenderer` for high-level declarative rendering.

**Behavioral details:**
- `EmmetRenderer` supports `isLoading`, error fallback, retry, and `onError` callback.
- Empty emmet input returns `null` render output (not hard failure).

**Acceptance criteria:**
- Consumer can choose integration level without rebuilding internals.

---

### FR-7: TypeScript and Package Surface
**Requirement:** Product MUST provide typed exports for core APIs, registry, actions, components, and pipeline types.

**Acceptance criteria:**
- Public entrypoint exports runtime and type symbols expected by host apps.

---

## 7) Non-Functional Requirements

### NFR-1: Security
- Sanitization MUST occur before hydration in all default flows.
- Sanitizer defaults MUST be deny-by-default with explicit allowlists.

### NFR-2: Reliability
- Pipeline stages SHOULD remain deterministic and composable.
- Recoverable errors should surface clearly to caller/UI.

### NFR-3: Performance
- Expensive transformation SHOULD be memoized in hook-based integration.
- Event handling SHOULD use delegation to avoid per-node listeners.

### NFR-4: Extensibility
- Registry design MUST allow custom tags/components without forking library.
- Stage functions SHOULD be exportable for advanced customization.

### NFR-5: Developer Experience
- Defaults should enable “works out of the box” behavior.
- Typed API contracts SHOULD minimize misuse.

---

## 8) User Journeys

### Journey A: LLM-to-UI Render
1. App receives user prompt.
2. LLM returns Emmet string.
3. App passes Emmet string to `EmmetRenderer`.
4. UI appears using default/custom components.

### Journey B: Action Feedback Loop
1. Rendered element includes `data-action="next"`.
2. User clicks element.
3. `onAction` receives action + optional payload.
4. App calls LLM with action context.
5. New Emmet string replaces old UI.

### Journey C: Platform Customization
1. Team creates `ComponentRegistry`.
2. Team registers proprietary components.
3. App injects registry via renderer options.
4. Emmet tags map to internal design-system components.

## 9) API-Level Requirements Snapshot

### Required Inputs
- Emmet string (non-empty for strict pipeline usage).

### Optional Inputs
- `onAction` callback.
- Custom `ComponentRegistry`.
- `className`, `isLoading`, `onError` (for `EmmetRenderer`).

### Outputs
- React tree or `null` depending on stage/input.
- Action callback invocations for user interaction events.
- Error objects surfaced via hook/component channels.

## 10) Constraints

1. Depends on Emmet grammar and parser behavior.
2. Depends on DOMPurify behavior in supported runtime environments.
3. Sanitizer allowlist limits what UI primitives can be emitted.
4. React peer dependency limited to v18/v19 range.

---

## 11) Risks and Mitigations

### Risk 1: Unsafe LLM output
- **Mitigation:** Mandatory sanitization stage before hydration.

### Risk 2: Unknown custom tags from LLM
- **Mitigation:** `UnknownTag` fallback + customizable registry.

### Risk 3: Event over-triggering in nested action nodes
- **Mitigation:** Explicit delegation logic documented and testable.

### Risk 4: Payload parse failures
- **Mitigation:** Non-throwing parse strategy returns Error payload.

---

## 12) Success Metrics (Suggested)

1. **Render success rate:** % of LLM outputs that render without fatal error.
2. **Security rejection coverage:** % of known XSS payload suite blocked.
3. **Interaction completion:** % of action clicks that produce expected callback.
4. **Token efficiency:** relative prompt+response token savings vs JSON schema workflows.
5. **Customization adoption:** % of consumers using custom registry entries.

---

## 13) Test and Validation Requirements

### Unit Tests Should Cover
- Emmet expansion valid/invalid paths.
- Sanitizer stripping of scripts/events/dangerous attributes.
- Hydration output and replace callback behavior.
- Registry CRUD semantics and fallback behavior.
- Action delegation and payload parsing edge cases.
- EmmetRenderer loading/error/retry behavior.

### Integration Tests Should Cover
- Full pipeline from Emmet input to interactive rendered output.
- LLM-style strings with mixed safe/unsafe constructs.

---

## 14) Release/Adoption Requirements

1. Build artifacts for ESM/CJS/types must publish cleanly.
2. Public README must keep Quick Start aligned with actual exports.
3. Prompt templates (`prompts/*.md`) should remain consistent with allowed tags/actions.

## 15) Open Product Decisions

1. Should allowlist include richer semantic elements (forms, tables, inputs)?
2. Should action dispatch support keyboard events and accessibility triggers?
3. Should unknown tags fail hard in production instead of rendering fallback?
4. Should payload parsing support non-JSON encodings for simpler authoring?
5. Should first-party adapter packages be added for major LLM providers?

### Remaining Gaps / Future Hardening

1. **CSP policy quality validation** is not enforced by library runtime (only presence warning).
2. **HTTP header-level checks** are not possible inside this UI library (host app responsibility).
3. **Accessibility-triggered interaction semantics** (e.g., keyboard activation that results in click in real browsers) can be expanded with higher-fidelity integration tests.

### Consolidation Opportunities (Optional)

1. Several sanitizer tests are intentionally repetitive for security regression confidence; can be table-driven later.
2. Some registry and renderer assertions can be merged to reduce duplication if test runtime becomes a concern.

### Conclusion

- **The previously identified high-priority edge cases are now covered by tests.**
- **Primary remaining work is host-app hardening (CSP headers/policy strictness), beyond this library’s runtime scope.**
